Decision pipeline

From application to decision in seconds

Every business that applies runs through an automated 4-stage pipeline. No human is consulted unless the system genuinely can't decide.

Submit
Business POSTs application. Validated with Pydantic, persisted as pending, Celery task enqueued.
Rule engine
5 deterministic checks: KYB completeness · Sanctions watchlist · Country risk · Credit threshold · Velocity
LLM analyst
Claude reads the full application and rule results, returns a 0–100 risk score, recommendation, and written rationale.
Decision
Auto-approved
Auto-declined
Human review

"The LLM augments and explains decisions, but never has unilateral authority over money movement. Deterministic rules hold veto power, and humans review on disagreement."

Core architectural thesis
Engineering decisions

Built the way financial infra should be

Every architectural choice reflects how real fintechs think about risk, compliance, and scale.

Deterministic rules first
Hard signals — sanctions hits, banned countries, repeated EINs — block before the LLM is ever consulted. Rules are config-driven in rules.yaml.
LLM adds judgment
Claude reads context the rules can't see. Returns structured JSON: risk score, recommendation, top concerns, and a rationale memo. Output is Pydantic-validated.
Fail-safe by design
LLM timeout or malformed output? The system defaults to needs_review — never auto-approve on failure. Errors escalate to humans, not to approvals.
Human review queue
Ambiguous cases land in a full ops console with the AI rationale, risk score, and concern chips visible — so reviewers make informed decisions fast.
Full observability
Prometheus metrics, OpenTelemetry traces, and a pre-built Grafana dashboard. Queue depth, decision latency, auto-decision rate — all instrumented.
Production-ready scale
Stateless API + stateless workers. Scale with --scale worker=N. Redis-backed idempotency, connection pooling, partition-ready audit log.

Tech stack

Purpose-built, not bolted on

Each technology earns its place. Nothing is included for buzzword coverage.

Python 3.12 Flask Celery Redis PostgreSQL Claude (Anthropic) Pydantic v2 SQLAlchemy 2 + Alembic Prometheus + Grafana OpenTelemetry Docker Compose GitHub Actions CI
Why async (Celery) instead of sync?
The API returns 202 instantly. LLM calls can take 2–5s — you can't make a user wait. Workers scale independently of the API tier.
Why not just trust the LLM?
LLMs hallucinate, time out, and can be prompt-injected. Deterministic rules hold veto power for hard signals — the LLM only gets to speak on ambiguous cases.
Why Redis for velocity checks?
Postgres is too slow for increment-then-expire under load. Redis atomic INCR + EXPIREAT gives microsecond-latency counters with no race conditions.
Why cache LLM responses?
Identical applications (duplicate filings, retries) shouldn't pay twice. A SHA-256 keyed Redis cache makes repeated assessments free and deterministic.
Why an append-only audit log?
Regulators require complete, immutable history. You can't update or delete audit records. The table is designed for range partitioning on created_at.
Why Pydantic on LLM output?
You can't trust free-form JSON from a language model in a financial context. Pydantic validates schema, types, and value ranges — malformed output triggers fail-safe.
Quick start

Running in 2 commands

Docker Compose brings up the full stack — API, workers, Postgres, Redis, Grafana, Prometheus.

Terminal 
# Clone and start
git clone https://github.com/RithikPorandla/gate-keeper.git
cd gate-keeper && docker compose up --build

# Seed with 12 synthetic applications
docker exec -it gatekeeper-api python scripts/seed.py

# Open the ops console
http://localhost:8000/dashboard    # metrics dashboard
http://localhost:8000/queue         # human review queue
http://localhost:3000               # Grafana
http://localhost:9090               # Prometheus

# Submit an application via API
curl -X POST http://localhost:8000/api/applications \
  -H "Content-Type: application/json" \
  -d '{"business_name":"Acme Corp","registration_number":"12-3456789",
     "country":"US","industry_code":"TECH","requested_spend_limit":50000,
     "applicant_name":"Jane Smith","applicant_email":"jane@acme.com",
     "mock_credit_score":720}'
